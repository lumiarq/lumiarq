<a name="introduction"></a>
## Introduction

Lumiarq is a full-stack TypeScript framework for building scalable, maintainable server-side applications. It is built around the **Porto** architecture pattern — a layered approach to organising code into cohesive, isolated modules with strict boundaries between concerns.

<a name="whats-in-the-box"></a>
## What's in the box

- **Module system** — self-contained bounded contexts with enforced layer boundaries
- **HTTP layer** — handler functions, Route DSL, route model binding, named routes
- **Business logic layer** — Actions, Tasks, Queries, and Events
- **Content loading** — markdown/frontmatter pipeline via `defineContentLoader`
- **CLI tooling** — `lumis` CLI with scaffolding, database, build, and dev commands
- **Auth starter** — session + JWT auth generated by a single command
- **Multi-target builds** — Node.js, Cloudflare Workers, or static HTML

<a name="key-concepts"></a>
## Key concepts

### Porto architecture

Porto organises application code into **modules**. Each module owns its domain completely — its routes, handlers, business logic, database schemas, and contracts. Modules communicate only through their public contracts, never by reaching into each other's internals.

```
src/modules/Billing/
├── module.ts                  # Module definition
├── index.ts                   # Public contract exports
├── contracts/                 # Types, DTOs, schemas
├── http/
│   ├── handlers/              # Request handlers
│   ├── routes/                # Route definitions
│   └── bindings/              # Route model bindings
├── logic/
│   ├── actions/               # Business logic (write side)
│   ├── tasks/                 # Side-effect workers
│   └── queries/               # Read-only queries
└── tests/                     # Module tests
```

### Convention-driven

Lumiarq uses conventions to reduce configuration. File naming determines route type (`*.web.ts` vs `*.api.ts`). Module discovery is automatic. Config files follow a consistent `satisfies` pattern.

### Full type safety

End-to-end TypeScript — from Zod-validated config files and request bodies, through typed DTOs and repository generics, to compile-time-safe route helpers.
